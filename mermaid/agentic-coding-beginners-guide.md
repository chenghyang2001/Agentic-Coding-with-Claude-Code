## 前言：你不孤單，而且你也不是最困惑的那個

讓我先告訴你一件事。

當我第一次聽到「Agentic Coding」這個詞的時候，我的大腦立刻聯想到的是某種昂貴的瑜伽課程，可能在峇里島舉辦，需要帶自己的瑜伽墊，以及一種奇特的「與宇宙程式碼連結」的冥想姿勢。

事實上，它不是。雖然學習過程中你可能確實需要不時閉眼深呼吸一下。

**Agentic Coding**（代理式程式設計）的意思，簡單說就是：讓 AI 不只是回答你的問題，而是真正替你「做事」。就像你僱用了一個極度聰明、從不疲倦、記憶力驚人的助理——除了偶爾它會以非常有創意的方式理解你的指令，然後做出讓你盯著螢幕沉默三秒的結果。

這本書（《Agentic Coding with Claude Code》）共有八個章節。你現在手上拿的這份指南，是我為你整理的入門手冊——從最簡單的開始，一路走到最複雜的山頂。沿途我會盡量讓你保持清醒，但無法保證你不會在某個地方停下來說「等等，AI 在呼叫另一個 AI？」

沒錯，它會。但那是第八章的事，別急。

## 難度 01 — 第八章：Output Styles
### 改變 AI 說話方式的神奇設定檔

**核心概念**：格式設定 | **技術難度**：★☆☆☆☆ | **需要寫程式**：完全不用

---

讓我們從最友善的地方開始。

Claude Code 有一個令人愉快的秘密：你可以告訴它「要用什麼方式回答你」。這聽起來很平常，但實際上非常強大。

想像你去同一家餐廳吃飯，每次點的是同樣的菜，但你可以要求：「請用精緻的法式擺盤」或「直接裝紙盒，我要帶走」或「給我畫一張這道菜的食材關係圖」。菜的內容完全一樣，但呈現方式截然不同。Claude Code 的 **Output Styles** 就是這個概念——你定義輸出的「格式」，AI 照著格式回答。

### 三種（加兩種）輸出風格

**yaml-concise（極簡 YAML 模式）**

這是給不喜歡廢話的人設計的。它的人生哲學是「說重點，其餘的請閉嘴」。一個 YAML 輸出大概長這樣：

```yaml
分析結果:
  問題: 函式太長
  嚴重程度: 中等
  建議: 拆成三個子函式
```

沒有前言，沒有客套，沒有「根據我的理解，您的問題可能涉及到...」。直接，清楚，效率至上。如果你是那種看到三頁說明書就頭疼的人，這個模式是你的心靈伴侶。

**retro-ascii-blog（復古終端機模式）**

這是為那些懷念 1980 年代、螢幕全是綠色文字的美好時代而設計的。黑底綠字，ASCII 裝飾框，一種「我正在駭入五角大廈」的愉悅感——儘管你實際上只是在問 AI 怎麼寫一個排序演算法。

```
╔══════════════════════════════╗
║  SYSTEM: ANALYSIS COMPLETE   ║
╚══════════════════════════════╝
> Loading results...
> ERROR FOUND: line 42
> RECOMMENDATION: refactor
```

它毫無實際用途，但看起來非常酷。有時候這就足夠了。

**mermaid-flowchart 和 mermaid-mindmap（視覺化圖表模式）**

這是本書額外新增的兩種模式，用於將 AI 的回答轉化為流程圖或心智圖，並自動匯出為 PNG 圖片。

是的，AI 不只回答你，還幫你畫圖，然後把圖存檔。科技真的進步了。你的祖父母不會相信這是真的。坦白說，一週前的你可能也不相信。

### 技術實作（真的很簡單）

你只需要在 `.claude/output-styles/` 目錄下建立一個 `.md` 設定檔，裡面描述你想要的格式，下次對話就能套用。沒有程式碼，沒有編譯，沒有依賴套件安裝失敗的悲劇，只有一個 Markdown 文字檔案。

這大概是整本書裡對新手最友善的章節，感謝作者的仁慈。

### 對你的工作有什麼意義

開會需要精準數字？用 yaml-concise。向老闆解釋架構決策？用 mermaid-flowchart。深夜除錯需要一點樂趣？用 retro-ascii-blog。同一個 AI，同樣的智能，無數種對話風格。

## 難度 02 — 第一章：Getting Started（入門基礎）
### 所有偉大旅程的起點都是「我要從哪裡下載？」

**核心概念**：基礎安裝與 CLAUDE.md | **技術難度**：★★☆☆☆ | **需要寫程式**：不需要，但要會用終端機

---

在我們踏入任何進階功能之前，讓我們先承認一件事：每個人都必須從這裡開始。

就連最資深的工程師，也有過那個盯著黑色終端機畫面、不確定自己是否做對了的瞬間。這是一種平等主義的困惑，無論你的 GitHub 上有多少個 star，都逃不過。

第一章處理的是基礎：安裝 Claude Code CLI、第一次執行、以及——這是最重要的——理解 **CLAUDE.md** 的概念。

### CLAUDE.md：給 AI 的員工手冊

把 CLAUDE.md 想像成一封你寫給 AI 助理的「入職指南」。當你在專案目錄下放一個 CLAUDE.md 檔案，每次 Claude Code 開始工作，它都會先讀這份文件。

你可以在裡面寫：「這個專案使用 Python 3.12，資料庫只能 SELECT 不能修改，commit 訊息要用繁體中文，所有函式必須有錯誤處理，如果你不確定，就先問我。」

AI 就會照著做。每一次，每一個 Session，不需要你每次都重新解釋規則。

這個設計的天才之處在於：它讓 AI 從一個「萬事從頭解釋」的模式，變成一個「我已經知道這個專案的規則和文化」的夥伴。就像請了一個新員工，但他第一天上班就已經讀完了所有內部文件，了解了所有不成文的規定，甚至知道哪些東西是絕對不能碰的。

沒有 CLAUDE.md 的 Claude Code，就像一個熱情但完全不了解你公司脈絡的新人——什麼都願意嘗試，但有時候做出來的東西讓你哭笑不得，需要花更多時間解釋「不是這個意思」。

## 難度 03 — 第三章（前半）：Custom Commands
### 懶惰，是程式設計師最偉大的美德

**核心概念**：自訂斜線指令 | **技術難度**：★★★☆☆ | **需要寫程式**：不需要，只要寫 Markdown

---

有一個著名的說法（出處不太確定，大概是某個週五下午決定「這件事我再也不要手動做第三遍」的工程師說的）：

> 「優秀的程式設計師是懶惰的。」

這裡的「懶惰」不是指不努力，而是指不願意重複做相同的事情——所以他們寫程式把這些事情自動化。

Custom Commands 就是這個精神的工具化體現。

### 斜線指令的魔法

你可以在 `.claude/commands/` 目錄下建立 Markdown 模板，定義屬於自己的斜線指令。書中的例子：

```
指令：/dad-joke
模板：Generate a dad joke about $arguments
```

然後你輸入 `/dad-joke 台灣的梅雨季`，Claude Code 就會生成一個關於台灣梅雨的父親笑話。（結果通常讓你嘆一口氣，然後默默傳給朋友——這大概就是「dad joke」真正的定義。）

### $arguments：讓指令活起來的變數

`$arguments` 這個變數讓你的指令變得動態。同一個模板，不同的輸入，不同的輸出。就像麵包機的烤模——形狀固定，但你可以放入任何你喜歡的麵糰。

你可以為常見的工作模式建立專屬指令：
- `/code-review` — 對當前修改進行程式碼審查
- `/write-test $arguments` — 為指定函式生成測試案例  
- `/explain $arguments` — 用最簡單的語言解釋技術概念

### 對你的工作有什麼意義

如果你每週都需要做相同格式的技術文件、相同結構的程式碼審查、相同風格的 commit 訊息，Custom Commands 讓這些事情變成一個指令的距離。

## 難度 04 — 第二章：HookHub 初版示範
### 一個網站的誕生，以及它如何說明「夠用就好」的哲學

**核心概念**：AI 輔助 Next.js 開發 + MVP 思維 | **技術難度**：★★★☆☆ | **需要寫程式**：需要，但有 AI 幫忙

---

第二章做了一件聰明的事：它不說理論，直接展示一個用 Claude Code 建造的真實產品——**HookHub**。

HookHub 是什麼？表面上，它是一個讓人們瀏覽、分享和發現開源 Claude Code Hooks 的社群平台。但更深一層，它是這整本書的縮影：一個用 AI 建造的平台，用來分享人們為 AI 建造的工具。

這裡有一點讓人頭暈的遞迴感，但讓我們不要想太深。

### 「夠用就好」是一種智慧，不是妥協

第二章展示的 HookHub 是 MVP（最小可行產品）版本：一個用 Next.js 建造的靜態展示網站，只有一個功能——展示 hooks。沒有用戶帳號，沒有評分系統，沒有評論功能，沒有 AI 推薦。

就，一個列表。

但這就夠了。因為在你確認「人們真的需要這個平台」之前，你不需要花六個月建造一個完整的社群系統。先讓真實的用戶告訴你他們需要什麼，再根據需求迭代。

這個思維——先建造核心，再根據回饋擴展——是第十章全端 HookHub 的重要前提。一個成功的複雜系統，是從一個可以運作的簡單系統演化而來的；而一個從一開始就複雜的系統，往往永遠完成不了。

## 難度 05 — 第四章：MCP 整合優化
### 給你的 AI 配備正確的工具箱，而不是所有的工具

**核心概念**：MCP 工具整合 + Context 成本控制 | **技術難度**：★★★★☆ | **需要寫程式**：需要設定 JSON

---

MCP（Model Context Protocol）是一個讓 Claude Code 連接外部工具和服務的協定。

讓我用比喻說明。假設你僱了一個超級助理（Claude Code），他非常聰明，但辦公室裡只有基本的文具。MCP 就是讓你可以在辦公室裡安裝專業工具的系統：要查 Gmail？裝個 Gmail 工具。要看 Google Calendar？裝個日曆工具。要控制瀏覽器？裝個瀏覽器工具。

沒有 MCP，Claude Code 是一個很聰明但活在真空中的大腦。有了 MCP，它可以即時搜尋網路、操作瀏覽器、讀取你的郵件和日曆。

### 但工具越多，問題越大

這是第四章真正要解決的問題。

如果你安裝了 20 個 MCP 工具，每次對話 AI 都要「讀懂」所有 20 個工具的說明文件，這會佔用大量的 context（也就是 AI 的思考空間），讓每次對話都變得又慢又貴。

就像你每次出門，都把家裡所有的工具都帶上——十七把螺絲起子、六種型號的鑽頭、一個你五年沒用過的焊接機——不是不行，但很不合理。

### 解決方案：按需載入

`--mcp-config` 旗標讓你可以為不同的任務建立不同的「工具箱設定檔」，只在需要的時候載入需要的工具。

寫程式的時候，只載入程式碼相關的工具。處理郵件的時候，只載入 Gmail 和 Calendar。這樣 AI 的注意力集中，速度快，成本低，效果好。

## 難度 06 — 第七章：Custom Agents
### 建立你的 AI 分身軍團，讓每個任務都有專屬的智能

**核心概念**：個性化子代理 | **技術難度**：★★★★☆ | **需要寫程式**：需要寫 YAML frontmatter

---

現在我們進入了真正有趣的領域。

Claude Code 允許你建立「自訂代理人」（Custom Agents）——本質上就是給 AI 貼上不同的人格標籤，讓同一個底層模型在不同情境下扮演不同的角色。

### 認識 CodeComedy Carl（卡爾哥）

書中展示的第一個自訂代理人叫做 CodeComedy Carl，一個有十五年程式開發經驗的資深工程師，同時也是一個業餘脫口秀演員。

他的程式碼審查，是這樣的：

「這個函式的邏輯就像我的感情生活——充滿了條件判斷，卻沒有一個清晰的出口，而且我懷疑它從來沒有在真實世界中被完整測試過。」

他有一套自創的效能評分系統，從「蝸牛步伐」（效能糟透了）到「火箭速度」（非常優化），讓程式碼審查從一件你會一直拖延的苦差事，變成一件你會想拿給同事看的有趣文件。

### 認識 Mermaid Diagram Generator

第二個代理人沒有個性，也不開玩笑。它的唯一任務是：把你描述的任何概念，轉化為 Mermaid 圖表，然後自動匯出為 PNG 圖片。

它不閒聊，不說廢話，只是非常專注且高效地把你的文字變成漂亮的視覺化圖形。在某些情況下，這種沉默的效率，比任何幽默都更令人感激。

### 技術實作

Custom Agents 定義在 `.claude/agents/` 目錄下的 Markdown 文件。每個文件頂部有 YAML frontmatter，定義：
- `name`：代理人的識別名稱
- `model`：使用哪個 Claude 模型（sonnet、haiku 等）
- `tools`：這個代理人可以使用哪些工具
- 文件正文：代理人的「個性描述」和行為準則

### 對你的工作有什麼意義

不同的任務需要不同的「心態」。程式碼審查需要嚴謹，創意發想需要開放，技術文件需要精確。與其每次都對同一個 AI 說「請你現在扮演一個嚴格的 code reviewer」，不如有一個隨時準備好的嚴格代理人，一呼即來。

## 難度 07 — 第三章（後半）：Hooks 系統
### 讓 AI 在你不注意的時候也在默默工作

**核心概念**：事件驅動自動化 | **技術難度**：★★★★★ | **需要寫程式**：需要，Python 腳本

---

讓我先說一個令人警醒的故事。

曾經有一個工程師，他的 Claude Code 在某個深夜執行了一系列操作，結果第二天早上他發現一些不應該被刪除的東西不見了。這不是 AI 的惡意，這是因為沒有任何機制來「監控和控制」AI 的行為。

Hooks 的誕生，部分就是為了讓這種故事不再發生。

### 什麼是 Hooks？

Hooks 是 Claude Code 在執行特定事件時，自動觸發的腳本程式。你可以把它想成「埋伏在工作流程中的守衛」。主要的 Hook 事件有：

**PreToolUse（工具使用前）**

在 Claude 使用任何工具之前執行。這是你的「安全警衛」。如果 Claude 想要執行一個看起來危險的指令，這個 Hook 可以在執行前暫停並問你確認。比如，如果 AI 突然想要刪除一個資料夾，PreToolUse 可以彈出一個「您確定嗎？」——就像 Windows 那個讓人又愛又恨的確認視窗，但這次是真的必要的。

**PostToolUse（工具使用後）**

在 Claude 完成一個操作後執行。可以用來記錄日誌、發送通知，或者（如書中的範例）播放一個悅耳的音效，讓你知道操作完成了。因為讓 AI 在做完一件事之後告訴你「做完了」，比讓你死盯著螢幕等待，要人性化得多。

**Notification（通知）和 Stop（停止）**

顧名思義，一個在 Claude 需要你注意時觸發，一個在整個任務完成時觸發。

書中的 `hooks-notification` 範例展示了：當 Claude Code 完成工作時，自動播放一個音效通知你——就像微波爐的「叮」聲，但這次宣告的是「你的 AI 助理已經完成了你交代的任務，請來取餐」。

### 技術實作

Hooks 在 `.claude/settings.json` 中設定，指向要執行的 Python 腳本。這些腳本可以做幾乎任何事：發送 Telegram 訊息、截圖存檔、查詢資料庫、播放音效，或者只是在日誌裡記錄「Claude 今天下午 3:42 使用了 Bash 工具」。

### 對你的工作有什麼意義

Hooks 讓 Claude Code 從「你問它答」的被動工具，變成「主動嵌入你工作流程」的自動化夥伴。它不再只是聊天機器人，而是一個與你的工作生命真正整合的系統，在背景默默守護，在必要時及時介入。

## 難度 08 — 第九章：Skills 系統
### 當 AI 開始呼叫 AI，宇宙發生了一點點顫抖

**核心概念**：Shell 腳本 + claude -p 遞迴代理 | **技術難度**：★★★★★ | **需要寫程式**：需要，Shell 腳本

---

我需要先讓你做好心理準備。

Skills 系統的核心概念是：Claude Code 可以在 Shell 腳本中執行 `claude -p` 指令，也就是說——**Claude Code，可以呼叫另一個 Claude Code，來完成子任務**。

AI，在呼叫 AI。

深呼吸一下。好了嗎？讓我繼續。

### git-pushing：一個 Skill 的完整示範

書中展示的 `git-pushing` Skill 是這個機制的完美例子。整個流程是這樣的：

1. 你說：「幫我推送變更」（就這三個字）
2. Skill 觸發 `smart_commit.sh` Shell 腳本
3. 腳本執行 `claude -p`，讓另一個 Claude 實例分析 git diff
4. 那個 Claude 實例生成一個符合 Conventional Commits 規範的 commit 訊息——以**海盜語氣**撰寫
5. 腳本自動暫存、提交、推送到遠端

對，**海盜語氣**。設定文件上明確寫著「生成海盜風格的 commit 訊息」。我沒有捏造這件事。你的 commit log 可能會出現「Arr, 吾等已修繕那惱人的登入漏洞，願伺服器永保穩健」之類的訊息。

這是完全真實的功能，存在於一本嚴肅的技術書籍中。我非常欣賞這個決定。

### Skills 的結構

每個 Skill 包含：
- `SKILL.md`：說明文件，定義觸發條件和工作流程
- Shell 腳本：實際執行工作的腳本，其中可以調用 `claude -p`

`.claude/skills/` 目錄下的每個子資料夾就是一個 Skill，整潔、獨立、可重複使用。

### 對你的工作有什麼意義

Skills 系統代表的是「複合智能」的概念：不是讓一個 AI 承擔所有工作，而是建立一個有組織的系統，讓不同的組件各司其職，協同完成複雜任務。

這是從「智能工具」到「智能工作流程」的關鍵躍升。而且，說實話，能讓 AI 以海盜語氣寫 commit 訊息，本身就已經是一個值得學習的理由。

## 難度 09 — 第十章：全端 HookHub
### 一個完整產品的誕生，以及 AI 記憶的重要性

**核心概念**：Spec 驅動開發 + Memory 系統 + 跨 Session 迭代 | **技術難度**：★★★★★★ | **需要寫程式**：需要全端開發能力

---

我們終於到達了山頂。

第十章展示了用 Claude Code 進行完整的全端應用開發——主角還是 HookHub，但這次是正式版本：Next.js + TypeScript + Tailwind CSS，一個真正可以部署的社群平台。

然而，這一章真正精彩的，不是 Next.js 的語法細節，而是兩個改變你對 AI 開發認知的核心概念。

### 概念一：Spec 規格書驅動開發

在寫任何程式碼之前，先建立 `memory/spec/CLAUDE.md`——一份完整的產品規格書，包含：

- **願景**：這個產品要解決什麼問題？
- **目標受眾**：誰會用它？
- **MVP 範圍**：能做什麼、明確不做什麼
- **資料模型**：資料長什麼樣子？
- **UI 需求**：介面要長什麼樣子？
- **技術選型決策**：為什麼選這個框架，不選那個？

有了這份文件，Claude Code 就有了「北極星」。它不會隨意做決定，不會突然把一個「顯示按鈕」過度工程化成一個需要三個微服務支撐的複雜系統，也不會在第五個 Session 的時候忘記你在第一個 Session 說過的「評分功能不在 MVP 範圍內」。

規格書驅動開發的本質，是讓你在開始之前先想清楚「你到底要什麼」——而不是邊做邊改、邊改邊增加範圍，直到這個專案變成一個誰也說不清楚它到底在做什麼的混沌體。

### 概念二：Memory 系統——讓 AI 記得昨天的決定

這是跨 Session 連貫性問題的解決方案。

當你關掉 Claude Code 再重新打開，它通常不記得上一次的對話。這對個人問答沒什麼關係，但對一個持續開發中的產品來說，「每次都從頭解釋一遍」不只浪費時間，還容易因為解釋的方式不同而產生前後不一致的決策。

`memory/` 目錄就是讓 AI「記住」重要決策的機制：
- `memory/spec/CLAUDE.md`：產品規格書
- `memory/frontend/CLAUD.MD`：前端開發風格指南（使用 TypeScript strict、early return、aria-label 無障礙標籤...）

每次新的 Claude Code Session 開始，它都會讀取這些文件，立刻進入狀況——就像一個每次上班前都認真讀交接紀錄的優秀員工。

### 兩個版本，一個關於迭代的故事

第十章最令人印象深刻的，是它展示了同一個 HookHub 應用在兩個不同 Claude Code Session 之間的演化：

**zealous-jemison（初版 Session）**
- 乾淨的靜態頁面
- 20+ 個精選 hooks，包含真實的 GitHub 星星數
- 功能完整，但視覺效果平淡
- 就是「能用」

**vigilant-feistel（進化版 Session）**
在初版的基礎上，新增了豐富的視覺效果：
- Logo 上下漂浮的動畫
- Hero 文字逐層淡入的效果
- 卡片 hover 時浮起的互動
- 「Browse Hooks」按鈕的橘色光暈脈動

這不是重做，也不是推翻，而是在已有基礎上的精煉。正是因為 Memory 系統保存了設計決策和產品規格，第二個 Session 的 Claude 才能知道「什麼已經做好了，什麼可以做得更好，什麼是不在範圍內的」。

**這就是 AI 輔助開發的正確姿態：** 不是讓 AI 一次性生成所有東西然後祈禱能用，而是透過清晰的規格書、系統性的 Memory 管理，以及每個 Session 的增量改進，建造真正可以持續演化的產品。

## 後記：你已經比你想像的更懂了

如果你讀到這裡，你大概已經從「Agentic Coding？某種瑜伽課嗎？」，進化到了：

> 「哦，所以 Output Styles 是 AI 的格式設定檔；Custom Commands 是可重複使用的指令模板；Hooks 是事件驅動的自動化守衛；Custom Agents 是個性化的子代理；Skills 是 Shell 腳本加上 AI 呼叫 AI 的遞迴魔法；而全端 HookHub 是這一切的完整體現——一個由規格書驅動、由 Memory 系統維持連貫性、在多個 Session 之間迭代演化的真實產品。」

這是一個相當顯著的進步。你值得給自己一點掌聲。

讓我用一個最後的比喻來總結這整個旅程：

Claude Code 就像一個可以被訓練、被配備、被客製化的智能工作夥伴。Output Styles 是它的說話方式，Custom Commands 是它的快捷鍵，Hooks 是它的行為準則，Agents 是它的不同人格，Skills 是它的複合能力，而全端開發則是它所有能力匯聚成真實產品的時刻。

每一層都建立在前一層的基礎上。每一個工具都是整個生態系統的一塊積木。而整個生態系統的頂點——HookHub——是一個由 AI 輔助建造的平台，讓社群可以分享他們為 AI 建造的工具。

如果這讓你頭暈，請喝一杯水。這是完全正常的。

然後，請去建造一些東西吧。
